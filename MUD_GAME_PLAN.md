# MUD 게임 2단계 - 기능 기획 및 프로젝트 구조 설계

## 1. 프로젝트 개요

이 문서는 C++ 초보 학습용 `LocalAIChat` 프로젝트를 바탕으로, 텍스트 기반 힐링 MUD 게임을 설계하기 위한 2단계 기획 문서이다. 이번 단계에서는 코드를 작성하지 않고, 나중에 C++ 코드로 옮기기 쉬운 기능 단위와 데이터 구조를 정리한다.

| 항목 | 설명 |
| --- | --- |
| 프로젝트 목적 | 로컬 LLM 채팅 프로그램을 확장해 텍스트 기반 생활형 MUD 게임을 설계한다. |
| 기존 프로젝트와의 관계 | 기존 채팅 입력, 명령어 처리, 기록 저장, 로컬 LLM 응답 흐름을 게임 입력과 NPC 대화의 기반으로 활용할 수 있다. |
| 게임 방향 | 전투보다 생활, 탐험, 수집, 주민과의 대화, 마을 분위기 형성을 중심으로 한다. |
| C++ 학습 연결 | 문자열 처리, 조건문, 함수, 클래스, vector 기반 목록 관리, 파일 저장, 상태 관리 학습으로 이어진다. |

동물의숲 스타일은 참고 방향일 뿐이며, 원본 게임의 대사, 세계관, 고유 설정을 그대로 복제하지 않는다. 비앙카(휘트니)를 모티브로 한 하얀 늑대 주인공은 학습용 기획에만 사용하고, 공개 배포 시에는 이름과 설정을 오리지널 캐릭터로 바꿀 수 있게 한다.

## 2. 게임 컨셉

전체 분위기는 조용하고 따뜻한 작은 마을에서 하루를 보내는 텍스트 기반 힐링 탐험 게임이다. 플레이어는 전투나 성장 수치보다 마을을 둘러보고, 주민과 대화하고, 작은 물건을 줍고, 부탁을 들어주며 마을 분위기를 느끼는 경험을 한다.

### 주인공 방향

- 하얀 늑대 캐릭터를 주인공으로 설정한다.
- 비앙카(휘트니)는 외형과 분위기의 학습용 모티브로만 참고한다.
- 공개 배포 가능성을 고려해 최종 이름, 말투, 배경은 오리지널 캐릭터로 바꿀 수 있게 설계한다.
- 주인공은 전사나 마법사가 아니라 새로 이사 온 주민 또는 마을 산책자에 가깝다.

### 마을과 장소의 분위기

- 마을은 작고 안전한 공간이다.
- 장소마다 다른 분위기와 수집 요소가 있다.
- 각 장소는 짧은 텍스트 설명으로 표현한다.
- 플레이어는 장소 설명을 읽고 이동, 대화, 수집을 선택한다.

### 생활형 콘텐츠 중심 구조

- 주민 NPC와 대화한다.
- 사과, 조개껍데기, 꽃, 나뭇가지 같은 작은 아이템을 줍는다.
- NPC의 가벼운 부탁을 들어준다.
- 아이템을 인벤토리에 보관한다.
- 집 꾸미기, 상점, 박물관, 계절, 날씨는 나중 단계 확장으로 남긴다.

## 3. 플레이어가 할 수 있는 행동

| 행동 | 예시 명령어 | 설명 |
| --- | --- | --- |
| 보기 | `look` | 현재 장소 설명, 아이템, NPC, 이동 가능한 방향을 확인한다. |
| 이동 | `north`, `go north` | 연결된 장소로 이동한다. |
| 말하기 | `talk 주민이름` | 같은 장소에 있는 NPC와 대화한다. |
| 아이템 줍기 | `take 사과` | 장소에 있는 획득 가능한 아이템을 인벤토리에 넣는다. |
| 아이템 사용 | `use 편지` | 아이템의 간단한 효과나 설명을 출력한다. |
| 인벤토리 확인 | `inventory` | 가지고 있는 아이템 목록을 본다. |
| NPC와 대화 | `talk guide` | 주민의 기본 대화, 힌트, 부탁을 확인한다. |
| 부탁 확인 | `quest` | 현재 받을 수 있거나 진행 중인 간단한 부탁을 확인한다. |
| 도움말 확인 | `help` | 사용할 수 있는 명령어 목록을 확인한다. |
| 게임 종료 | `quit` | 게임을 종료한다. |

초기 구현에서는 명령어 수를 너무 늘리지 않는다. 3단계에서는 `look`, 이동, `help`, `quit` 정도만 먼저 구현하고, 대화와 아이템은 작은 예시로 제한하는 것이 좋다.

## 4. 방 또는 장소의 개념

MUD 게임에서 방은 플레이어가 머무를 수 있는 장소 단위이다. 실제 방만 뜻하는 것이 아니라 광장, 숲길, 해변 같은 모든 장소를 방으로 볼 수 있다.

### 장소가 가질 정보

| 정보 | 설명 | C++로 옮길 때 예상 형태 |
| --- | --- | --- |
| 방 이름 | 장소 이름 | `std::string name` |
| 방 설명 | 플레이어에게 보여줄 설명 | `std::string description` |
| 연결된 방향 | 북쪽, 남쪽 등 이동 가능 방향 | 간단한 단계에서는 조건문 또는 vector로 처리 가능 |
| 방 안의 아이템 | 현재 장소에서 주울 수 있는 아이템 | `std::vector<Item>` |
| 방 안의 NPC | 대화 가능한 주민 | `std::vector<Npc>` |
| 방 상태 | 방문 여부, 아이템 획득 여부 등 | bool 값 또는 간단한 상태 문자열 |

### 예시 장소

| 장소 | 분위기 | 포함 가능 요소 |
| --- | --- | --- |
| 마을 광장 | 작은 게시판과 벤치가 있는 중심지 | 마을 안내자, 꽃, 게시판 |
| 비앙카의 집 | 흰색과 파란색 장식이 있는 아늑한 집 | 편지, 작은 장식품 |
| 숲길 | 나무 사이로 햇빛이 들어오는 길 | 사과, 나뭇가지, 친절한 주민 |
| 해변 | 조용한 파도 소리가 들리는 장소 | 조개껍데기, 해변 주민 |
| 상점 앞 | 아직 열리지 않은 작은 가게 앞 | 상점 주인, 안내 표지판 |
| 주민의 집 | 주민의 개성이 드러나는 집 | NPC, 부탁 힌트 |
| 작은 연못 | 꽃과 물소리가 있는 쉼터 | 꽃, 작은 보물 |

## 5. 이동 명령어

초기 이동은 네 방향만 사용한다.

- `north`
- `south`
- `east`
- `west`

나중에는 `go north`, `move east` 같은 확장 명령어를 추가할 수 있다.

### 이동 성공 흐름

1. 플레이어가 `north`를 입력한다.
2. 현재 장소에 북쪽 연결이 있는지 확인한다.
3. 연결된 장소가 있으면 현재 위치를 변경한다.
4. 새 장소 이름과 설명을 출력한다.
5. 새 장소의 아이템, NPC, 이동 가능 방향을 간단히 보여준다.

### 이동 실패 상황

- 연결된 방향이 없다.
- 아직 갈 수 없는 장소다.
- 명령어가 잘못 입력되었다.

예시 출력:

```text
그쪽으로는 갈 수 없습니다.
다른 방향을 선택해 보세요. 이동 가능 방향: north, east
```

### look 명령어

`look`은 현재 장소의 정보를 다시 보여주는 명령어이다. 이동 후에도 자동으로 `look`과 비슷한 정보를 출력하면 초보자가 게임 상태를 이해하기 쉽다.

## 6. 아이템과 인벤토리

아이템은 장소에서 발견하고 인벤토리에 보관할 수 있는 작은 물건이다. 초기에는 판매 가격이나 복잡한 효과 없이 이름, 설명, 획득 가능 여부 정도만 둔다.

### 아이템 정보

| 정보 | 설명 |
| --- | --- |
| 아이템 이름 | 예: 사과, 조개껍데기 |
| 아이템 설명 | 플레이어에게 보여줄 짧은 설명 |
| 획득 가능 여부 | 주울 수 있는지 여부 |
| 사용 가능 여부 | `use` 명령어로 반응할지 여부 |

### 생활형 아이템 예시

| 아이템 | 설명 | 사용 예시 |
| --- | --- | --- |
| 사과 | 숲길 아래에 떨어진 빨간 사과 | 먹거나 주민에게 줄 수 있는 후보 |
| 조개껍데기 | 해변에서 주운 작은 조개껍데기 | 수집품 또는 판매 후보 |
| 꽃 | 연못 근처에 핀 작은 꽃 | 선물 또는 장식 후보 |
| 나뭇가지 | 숲길에서 주운 가벼운 나뭇가지 | 제작 시스템 후보 |
| 편지 | 누군가에게 전달해야 할 편지 | 퀘스트 아이템 후보 |
| 작은 장식품 | 집 꾸미기에 쓸 수 있는 물건 | 나중 단계 꾸미기 후보 |

### 인벤토리 저장 방식

초기에는 `std::vector<Item>`처럼 목록으로 저장하는 방식을 생각할 수 있다. 3단계 최소 구현에서는 복잡한 `Item` 클래스 없이 문자열 목록으로 시작해도 된다.

나중에 확장할 수 있는 기능:

- 아이템 판매
- NPC에게 선물하기
- 집 또는 개인 공간 꾸미기
- 박물관에 기증하기
- 수집 도감 만들기

## 7. NPC와 대화

NPC는 마을 분위기를 만드는 핵심 요소이다. 초기에는 NPC마다 이름, 설명, 기본 대화, 간단한 힌트 정도만 가진다.

### NPC 정보

| 정보 | 설명 |
| --- | --- |
| NPC 이름 | 주민 이름 또는 역할 이름 |
| NPC 설명 | 외형이나 분위기 설명 |
| 기본 대화 | 규칙 기반으로 출력할 짧은 대화 |
| 성격 또는 말투 | 친절함, 조용함, 활발함 등 |
| 퀘스트 힌트 | 부탁이나 수집 목표를 알려주는 문장 |
| 장소 설명 | 현재 장소에 대한 주민의 한마디 |

### 예시 NPC

| NPC | 역할 | 예시 대화 방향 |
| --- | --- | --- |
| 친절한 주민 | 마을 분위기 소개 | “오늘은 숲길에 사과가 많이 떨어졌대.” |
| 상점 주인 | 상점 기능의 예고 | “가게는 아직 준비 중이지만, 나중엔 물건을 사고팔 수 있을 거야.” |
| 해변을 산책하는 주민 | 해변 수집 힌트 | “파도 사이에 예쁜 조개가 숨어 있을지도 몰라.” |
| 마을 안내자 | 튜토리얼 담당 | “처음이라면 look과 help를 입력해 봐.” |

구체적인 대사와 퀘스트 내용은 예시 수준으로만 작성한다. 원본 게임의 고유 대사나 설정을 그대로 사용하지 않는다.

## 8. 로컬 LLM을 활용한 AI NPC 응답

기존 LocalAIChat은 규칙 기반 응답과 Ollama 로컬 AI 응답 fallback 구조를 가진다. MUD 게임에서는 이 구조를 NPC 대화에 활용할 수 있다.

### 규칙 기반 응답과 로컬 LLM 응답의 차이

| 방식 | 장점 | 주의점 |
| --- | --- | --- |
| 규칙 기반 응답 | 예측 가능하고 구현이 쉽다 | 대화가 단순해질 수 있다 |
| 로컬 LLM 응답 | NPC 대화가 자연스럽다 | 게임 규칙을 벗어날 수 있어 제한이 필요하다 |

### AI NPC가 답변할 수 있는 범위

- 현재 장소 분위기 설명
- NPC 성격에 맞는 짧은 대화
- 플레이어가 가진 아이템에 대한 반응
- 진행 중인 부탁에 대한 힌트
- 게임 명령어 안내

AI NPC가 하면 안 되는 것:

- 존재하지 않는 장소로 이동시키기
- 플레이어에게 없는 아이템을 있다고 말하기
- 완료하지 않은 퀘스트를 완료했다고 말하기
- 전투, 레벨업, 스킬 같은 이번 게임 방향과 맞지 않는 시스템을 만들어내기
- 원본 게임의 고유 대사나 세계관을 따라 하기

### 프롬프트에 포함할 정보

AI NPC에게 보낼 프롬프트에는 아래 정보를 짧게 포함할 수 있다.

```text
너는 텍스트 기반 힐링 MUD 게임의 NPC이다.
원본 게임의 고유 대사나 설정을 복제하지 말고, 오리지널 마을 주민처럼 답한다.
현재 장소: 숲길
장소 설명: 나무 사이에 햇빛이 들어오고 사과가 떨어져 있다.
NPC 이름: 친절한 주민
NPC 성격: 차분하고 다정함
플레이어 인벤토리: 사과, 조개껍데기
퀘스트 상태: 편지를 아직 전달하지 않음
플레이어 질문: 여기서 뭘 할 수 있어?
규칙: 짧게 답하고, 실제 게임 상태를 바꾸지 않는다.
```

### 연결 실패 시 fallback

Ollama가 꺼져 있거나 응답 추출에 실패하면 규칙 기반 대화로 fallback한다. 예를 들어 NPC별 기본 대화 문장을 출력하면 된다.

```text
친절한 주민: 오늘은 마을 광장을 천천히 둘러보는 것도 좋겠어.
```

## 9. 게임 상태 저장 여부

MUD 게임에서는 대화 기록뿐 아니라 게임 진행 상태를 저장할 수 있다. 다만 초기 단계에서는 너무 많은 정보를 저장하지 않는다.

### 저장할 정보 후보

| 정보 | 설명 |
| --- | --- |
| 현재 플레이어 위치 | 마지막으로 있던 장소 |
| 인벤토리 | 가지고 있는 아이템 목록 |
| 방문한 방 | 한 번이라도 방문한 장소 |
| NPC 대화 진행 상태 | 특정 NPC와 대화했는지 여부 |
| 부탁 또는 퀘스트 상태 | 시작, 진행 중, 완료 같은 간단한 상태 |
| 획득한 수집 아이템 | 이미 주운 아이템을 다시 나오지 않게 하기 위한 정보 |

### 저장하지 않아도 되는 정보

- 매번 랜덤으로 바뀌어도 되는 장소 묘사
- 아직 구현하지 않은 날씨, 계절, 상점 재고
- 플레이어가 단순히 본 도움말 기록
- AI가 생성한 임시 대화 전체 내용

### 저장 방식

초기에는 텍스트 파일로 저장할 수 있다.

```text
location=town_square
inventory=apple,shell,flower
visited=town_square,forest_path,beach
quest_letter=started
```

나중에는 JSON 형식으로 확장할 수 있다. 단, 초보 단계에서는 복잡한 JSON 파서를 직접 만들기보다 단순한 `key=value` 형식부터 시작하는 것이 좋다.

## 10. 나중에 네트워크 게임으로 확장할 때 고려할 점

이번 단계에서는 네트워크 코드를 작성하지 않는다. 다만 나중에 여러 플레이어가 접속하는 MUD로 확장하려면 아래 사항을 고려해야 한다.

| 고려할 점 | 설명 |
| --- | --- |
| 여러 플레이어 접속 | 각 플레이어의 이름, 위치, 상태를 따로 관리해야 한다. |
| 플레이어별 위치와 인벤토리 | 한 플레이어의 이동이나 아이템이 다른 플레이어와 섞이면 안 된다. |
| 채팅 동기화 | 같은 방에 있는 플레이어에게만 대화를 보여줄지 정해야 한다. |
| 방 상태 공유 | 누가 아이템을 주웠을 때 다른 플레이어에게도 사라질지 정해야 한다. |
| 주민 NPC 대화 동기화 | NPC가 여러 플레이어와 동시에 대화할 때 상태 충돌을 막아야 한다. |
| 서버와 클라이언트 분리 | 게임 상태는 서버가 관리하고, 클라이언트는 입력과 출력만 담당할 수 있다. |
| 동시성 문제 | 동시에 같은 아이템을 줍는 상황 같은 충돌을 처리해야 한다. |
| 저장 데이터 충돌 | 여러 플레이어의 저장 정보가 서로 덮어써지지 않게 해야 한다. |

네트워크 확장은 C++ 기초 학습 후반 단계의 큰 주제로 남긴다.

## 11. 3단계에서 구현할 최소 기능 제안

다음 단계에서는 아직 큰 게임을 만들지 말고, 작게 실행 가능한 MUD 뼈대만 구현하는 것이 좋다.

### 추천 최소 범위

1. 방 3개
   - 마을 광장
   - 숲길
   - 해변
2. 이동 명령어
   - `north`
   - `south`
   - `east`
   - `west`
3. `look` 명령어
   - 현재 장소 이름과 설명 출력
   - 이동 가능 방향 출력
4. `help` 명령어
   - 사용 가능한 명령어 안내
5. `quit` 명령어
   - 게임 종료
6. 간단한 아이템 2개
   - 사과
   - 조개껍데기
7. NPC 1명
   - 마을 안내자
8. 규칙 기반 대화 1개
   - `talk guide` 입력 시 짧은 안내 출력

### 3단계에서 아직 하지 않을 것

- 실제 AI NPC 연결 코드 추가
- 파일 저장
- 복잡한 퀘스트 시스템
- 상점, 박물관, 계절, 날씨
- 집 꾸미기
- 네트워크 기능
- 전투, 레벨업, 직업, 스킬 시스템

3단계의 목표는 “플레이어가 텍스트 명령어로 장소를 보고 이동할 수 있다”를 실행해서 확인하는 것이다.
